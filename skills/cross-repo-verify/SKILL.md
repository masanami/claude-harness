# クロスリポジトリ依存の確証

実装が**他リポジトリのコードの挙動**（API の副作用・導出・判定ロジック等。例:「A を同期すれば B が作られる」）に依存する仮定を含む場合に、その仮定を**依存先リポジトリの実コード（判定箇所・書き込み箇所）で確証**する手順です。

クロスリポジトリ依存が無い場合はこのスキルを使用しない。

## 手順

### 1. コミット SHA の解決

**取得前にコミット SHA を固定する**。ブランチ名のまま取得すると、取得中にブランチが進んだ場合に本文と記載する根拠 SHA がずれ得るため、先に対象ブランチの HEAD SHA を解決する:

```bash
gh api "repos/{owner}/{repo}/commits/{branch}" --jq '.sha'
```

### 2. 依存先実コードの取得

依存先の clone は不要。Step 1 で解決した SHA を `ref` に指定して `gh api` で対象ファイルの本文を read-only に取得し、判定箇所・書き込み箇所のコードを直接確認する:

```bash
gh api -H "Accept: application/vnd.github.raw+json" "repos/{owner}/{repo}/contents/{path}?ref={SHA}"
```

### 3. 根拠の記載形式

根拠は**Step 1 で解決したコミット SHA を記載する**。ref の省略やブランチ名のままの記載は不可とし、`repo/file:line` だけでは後から別版のコードが根拠扱いになり得るため、`repo@コミットSHA/file:line` 形式で統一する。

### 4. 確証結果テーブル

確証結果は以下の形式で設計成果物または PR 本文に記載する:

```text
## クロスリポジトリ依存の確証

| 仮定 | 確証結果 | 根拠（repo@コミットSHA/file:line） |
|------|---------|--------------------------|
| {API X を呼ぶとテーブル B が導出される} | ✅ 確証 / ⚠️ 未確証 | {owner/repo}@{コミットSHA}/{path}:{行} |
```

後述「確証の独立再照合」によって ✅ 確証から格下げされた行も、テーブル上の表記は既存の2値のまま「⚠️ 未確証」に統一する（新しい記号は追加しない）。格下げされた行だと分かるよう、テーブルの下に `- {仮定}: 独立再照合により未確証へ格下げ（reason: {懐疑者の reason}）` の形式で注記を列挙し、同内容を PR 本文にも明示する。

## 規律

- **自リポジトリのモックテストが green であることは、仮定の検証とみなさない**。モックは「X を呼んだこと」しか検証できず、「X を呼んだら何が起きるか」は検証対象外
- 確証できない場合は「**未確証の仮定**」として明示し、仮定が誤っていた場合の影響（実装が no-op になる等）を添える

## 確証の独立再照合

確証結果テーブルが実際に生成された場合（行が1行以上存在する場合）にのみ、テーブルの行ごとに `claude-harness:attestation-verifier` を Task ツールで独立に spawn し、確証の真正性を反証させる。**確証した本人（このスキルを実行している自分自身）が取得済みのファイル内容を懐疑者に代弁させて信用させてはならない** — 懐疑者は根拠に記載された SHA から自分で対象ファイルを取得し直す前提で設計されている。テーブルの行が複数ある場合は並列 spawn してよい。

各行の spawn には以下を渡す:
- 仮定
- 確証結果（✅ 確証 / ⚠️ 未確証）
- 根拠（`{owner/repo}@{SHA}/{path}:{行}`）

呼び出し例:

```text
Task(
  subagent_type: 'claude-harness:attestation-verifier',
  prompt: "以下のクロスリポジトリ依存確証の1行を独立に反証してください。\n仮定: {仮定}\n確証結果: ✅ 確証\n根拠: {owner/repo}@{コミットSHA}/{path}:{行}"
)
```

返ってきた verdict が `not_supported`（引用は仮定を支持しない。反証成立）または `unverifiable`（判定不能）の行は、確証結果テーブルの ✅ 確証を外し「⚠️ 未確証の仮定」へ**格下げ**する。格下げした行は PR 本文に明示し、懐疑者の `reason` を添える。`supported`（引用は仮定を支持する）verdict の行は、元の ✅ 確証のまま確証結果テーブルに残してよい。

**発動条件**: 本節はクロスリポジトリ依存の確証結果テーブルが実際に生成された場合にのみ適用する。クロスリポジトリ依存が無くテーブル自体が存在しない通常ケースでは、本節の手順は一切実行しない。
