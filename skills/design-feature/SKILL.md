# 機能設計（feature-level design）

要件チケット（またはドキュメント）をもとにコードベースを分析し、**1機能スコープの設計ドキュメント**を作成します。アーキ決定・IF・データモデル・実装計画を1本にまとめ、後続の `/create-ticket` で実装チケット分解、`feature-implementer` でチケット単位の実装に進める形式に揃えます。

> このスキルは「機能単位の設計」専用です。**個別の設計判断の記録**（背景・選択肢・帰結）が目的の場合は ADR（Architecture Decision Record）形式のスキルや別ドキュメントを検討してください。

---

## 入力

親Issue番号またはパス: $ARGUMENTS

| 形式 | 解釈 |
|---|---|
| 数値・`#数値` | 親Issue番号として `gh issue view` で取得 |
| パス・ファイル | 該当ファイル/ディレクトリ配下の `.md` を読み込み |
| なし | エラー、ユーザーに入力を促す |

---

## 手順

### Step 1: プロジェクト理解

`CLAUDE.md` を読み、以下を把握する:

- 技術スタック・ディレクトリ構成
- 開発原則・設計規約
- ドキュメントマップ（既存設計ドキュメント・テンプレートパス）

### Step 2: 要件の読み込み

入力に応じて要件を取得:

- 親Issue番号: `gh issue view {番号} --json title,body,labels,number`。本文の要件・完了条件・受入基準を把握する。リンクされた要件ドキュメントがあれば併せて読む
- パス: 該当ファイル / 配下の `.md` を読み込む

### Step 3: コードベース分析

要件に関連する既存コードを Glob/Grep で分析する:

1. **関連モジュールの特定** — 影響範囲のディレクトリ・ファイルを列挙
2. **既存パターンの把握** — 類似機能の実装パターンを参照（同じ層の既存コード）
3. **依存関係の調査** — 上流・下流のモジュールを特定
4. **技術的制約の確認** — フレームワーク制約・既存APIとの整合性

### Step 4: 設計ドキュメント作成

テンプレートを次の優先順位で選ぶ:

1. `CLAUDE.md` に設計ドキュメントテンプレートのパスが指定されていればそれ
2. なければ `skills/design-feature/templates/design-doc.md`

`{slug}` 生成:

- 親Issueのタイトルから機能名を抽出
- 大文字→小文字、スペース・記号→ハイフン、連続ハイフン圧縮、先頭末尾ハイフン除去
- 既存と重複なら `-2`, `-3` の連番

保存先: `docs/design/{slug}.md`（`CLAUDE.md` で別パスが指定されていればそれ）

内容に必ず含める:

- **アーキテクチャ決定** — 実装方針と理由、変更対象モジュール、既存コードとの関係
- **IF / API** — モジュール間の境界となるインターフェース契約（該当する場合）
- **データモデル** — DBスキーマ変更・データ構造（該当する場合）
- **実装計画** — チケット分解案、依存関係、リスク

該当しないセクションは**セクションごと削除**して可（埋めるための水増しは禁止）。

### Step 5: ユーザー確認

設計ドキュメントの内容をユーザーに提示し、フィードバックを求める。指摘があれば修正して再提示する。

## 完了報告

```text
## 設計完了

- 設計ドキュメント: docs/design/{slug}.md
- 親チケット: #{番号}

次のステップ:
- 実装チケットへの分解: /create-ticket
- 実装チケットが揃ったら: /para-impl <実装チケット番号...>
```

## このスキルの責務外

- 要件定義（`/define-requirements`）
- 実装チケットへの分解（`/create-ticket`）
- 個別の設計判断記録（ADR が必要なら別スキル）
- 実装計画→TDD実装（`feature-implementer` / `/tdd-impl`）
