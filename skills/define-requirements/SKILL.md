# 要件定義

ユーザーとの対話を通じて要件を具体化し、要件ドキュメント（`.md`）と GitHub Issue（要件チケット）を作成します。

生成した要件ドキュメントは要件チケットとして登録され、後続の `/design-feature`・`/create-ticket`・`/para-impl` の入力として使えます。

---

## 入力

テーマや概要: $ARGUMENTS

---

## 手順

### 1. プロジェクト理解

`CLAUDE.md` を読み、以下を把握:

- 技術スタック
- ディレクトリ構成
- 既存機能・モジュール
- ドキュメントの配置ルール（`docs/` 配下の構造など）

### 2. 要件ヒアリング

`$ARGUMENTS` が指定されている場合はそれをテーマとして使用し、不足情報をユーザーに確認する。指定がない場合はユーザーに要件のテーマを質問する。

以下の観点を確認（すべて必須ではなく、機能に応じて取捨選択する）:

- **何を作りたいか**: 機能の概要
- **なぜ必要か**: 背景・目的
- **誰が使うか**: ユーザーストーリー
- **何ができればよいか**: 機能要件（具体的な振る舞い）
- **制約はあるか**: 非機能要件（パフォーマンス、セキュリティ等）、技術的制約
- **画面・APIはあるか**: UI/APIの設計指針（該当する場合）
- **どうなれば完了か**: 受入基準

> **ヒアリングのコツ**: 一度に全項目を聞くのではなく、概要→詳細の順で対話的に深掘りする。ユーザーの回答から推測できる項目は提案して確認を取る。

### 3. クリティカル設計の調査・意思決定

要件に**クリティカル領域**が含まれるかを判定し、含まれる場合は **このフェーズで意思決定までまとめる**。実装フェーズに人間ゲートを残さないことで、後続の `/para-impl` で複数チケットを中断なく並列実行できるようにする（戦略§3.4）。

#### 3-1. クリティカル領域の識別

要件から次の領域への影響有無を確認する:

| 領域 | 例 |
|------|-----|
| セキュリティ | 認証フロー、認可ポリシー、入力バリデーション |
| 機密データ・個人情報 | 個人情報の保管・転送・ログ出力ポリシー |
| 外部システム連携 | 外部API契約、リトライ・タイムアウト方針、リクエスト署名 |
| DBスキーマ変更 | テーブル追加・カラム変更・インデックス設計・データ移行 |

該当しない場合はこのステップをスキップする。

#### 3-2. 既存パターン・制約の調査

該当領域ごとにコードベース・既存設計を Glob/Grep で調査する:

- 既存の認可モデル / 暗号化方式 / 外部APIクライアント / マイグレーション方式 など
- 関連する設計ドキュメント（CLAUDE.md ドキュメントマップから辿る）
- フレームワークの推奨パターン・制約

#### 3-3. 設計選択肢の提示と意思決定

ユーザーに**選択肢と推奨案**を提示し、決定する:

```text
## クリティカル設計の意思決定

### {領域名 例: 認可モデル}

**選択肢**:
- A: {案A} — {pros/cons}
- B: {案B} — {pros/cons}

**推奨**: {案X}（理由: {既存パターンとの整合・実装コスト・拡張性 等}）

採用案を選んでください。
```

ユーザーが選んだ案を確定する。必要なら複数領域について順に決定する。

#### 3-4. 決定事項の文書化

決定内容を要件ドキュメントの **「## クリティカル設計決定」** セクションに記述する（次ステップでドキュメント化）。後続の `/design-feature` および `feature-implementer` はこの決定に従い、独自判断で逸脱しない。

### 4. テンプレート読み込み

以下の優先順位でテンプレートを読み込む:

1. `CLAUDE.md` に要件ドキュメントテンプレートのパスが指定されている場合はそれを使用
2. プロジェクトルートに `docs/templates/requirements-doc.md` が存在する場合はそれを使用
3. いずれもない場合はデフォルトの `skills/define-requirements/templates/requirements-doc.md` を使用

> **CLAUDE.md での指定例**: ドキュメントマップやテンプレートセクションに `要件ドキュメントテンプレート: docs/templates/requirements-doc.md` のように記載する。

### 5. 要件ドキュメント作成

テンプレートに沿って要件ドキュメントを作成する。

- **保存先**: `docs/requirements/{slug}.md`
  - `CLAUDE.md` に要件ドキュメントの保存先が指定されている場合はそれに従う
  - `{slug}` は機能名をケバブケースにしたもの（例: `user-authentication`）
- **内容**: ヒアリング結果をテンプレートの各セクションに記述
  - 該当しないセクションは「（該当なし）」ではなくセクションごと削除する
  - 機能要件・受入基準はチェックボックス形式で記述する

### 6. GitHub Issue 作成

`skills/create-ticket/SKILL.md` の要件チケット手順に従い、GitHub Issue を作成する。

- テンプレート: `skills/create-ticket/templates/requirement-ticket.md`
- Issue 本文に以下を含める:
  - 要件ドキュメントへのリンク: `要件ドキュメント: docs/requirements/{slug}.md`
- ラベル: `requirement`（またはプロジェクトの慣習に従う）

### 7. 完了報告

以下の形式で報告する:

```
## 要件定義 完了

- 要件ドキュメント: `docs/requirements/{slug}.md`
- GitHub Issue: #{番号}

次のステップ:
# 1. 機能設計
/design-feature #{番号}

# 2. 設計を実装チケットに分解
/create-ticket

# 3. 実装チケットが揃ったら並列実装
/para-impl <実装チケット番号...>
```
