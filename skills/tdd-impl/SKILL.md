# TDD実装（Kent Beck / t-wada 流）

設計成果物（実装計画）に従って、**テスト駆動開発（TDD）**を **「リスト → レッド → グリーン → リファクタ」** の規律で進める手法を提供します。

> このスキルは「設計が確定した後」の実装フェーズに特化しています。要件理解・設計成果物の出力・必須ゲート通過・セルフレビュー等は呼び出し元（例: `feature-implementer` エージェント）の責務です。

---

## 前提

呼び出し元から以下が確定して渡されている前提:

- **対象仕様**: チケットの完了条件・受入基準（Acceptance Criteria）
- **実装計画**: 何をどの順に作るかの方針
- **プロジェクトの規約**: テストフレームワーク、命名規則、ディレクトリ構成（CLAUDE.md 経由）

これらが揃っていない場合は呼び出し元に差し戻す（このスキル内では新たに要件・設計を作らない）。

---

## TDD の正式な定義（Kent Beck）

TDD は次の5ステップを繰り返すワークフローである:

1. **テストリスト**（網羅したいテストシナリオの一覧）を書く
2. リストから **ひとつだけ** 選び、実行可能なテストコードに翻訳し、**失敗することを確認**する（Red）
3. プロダクトコードを変更し、いま書いたテスト（とそれまでのすべて）を **成功させる**（Green）
4. 必要に応じて **リファクタリング** を行い、設計を改善する
5. テストリストが空になるまでステップ 2 に戻って繰り返す

> **本質は「リスト → レッド → グリーン → リファクタ」**。テストリストが明示的に存在することで、設計のやりすぎ・やらなさすぎの両方を回避できる。

### TDD ではないもの

以下は TDD ではない。混同しない:

- テストを先に **たくさん** まとめて書くこと（→ 単なる「テストファースト」）
- リファクタリングなしでテストとコードだけ書くこと（→ 単なる「自動テスト」）
- 設計なしにいきなりコードを書き始めること

---

## Phase 1: テストリストの作成（分析フェーズ）

実装に入る前に、**期待される振る舞いを網羅した「テストリスト」** を作る。実装判断はこの段階では持ち込まない。

- 完了条件・受入基準を「テストとして検証できる粒度」に分解する
- 正常系・異常系・境界値・特殊条件を含める
- 例: 「正常系はこう、タイムアウト時はこう、キーが無い時は…」
- 1項目1テストに対応するよう、過不足なくリストアップする

### リストの順序付け

次の優先順位で並べ替える:

1. **テスト容易性が高い**: 副作用・依存が少なく、書きやすいもの
2. **重要度が高い**: 完了条件の核心に近いもの
3. **不確実性が高い**: 動かしてみないと分からないもの

> 1〜3 をすべて満たすものから着手すると、フィードバックが速く得られる。

### 出力フォーマット例

```text
## テストリスト

- [ ] 空文字列を渡すと空配列が返る（正常系・最小）
- [ ] 1要素のリストを渡すとそのまま返る（正常系）
- [ ] 重複を含む3要素のリストで重複除去される（正常系・主目的）
- [ ] null を渡すと TypeError（異常系）
- [ ] 1万件の入力で性能劣化しないこと（性能・非機能）
```

リストは作業中に追加してよい（実装中に気付いた観点）。ただし**実装と同時に追加しない**（先にリストへ書いてから着手する）。

---

## Phase 2: Red — 失敗するテストを「ひとつ」書く

リストから 1 項目だけ選び、テストコードに翻訳する。

### 守ること

- **Arrange-Act-Assert** で構造化する（準備・実行・検証）
- **意味のあるアサーション**を必ず置く（`expect`/`assert` が無いテスト・常に真の検証は禁止）
- **テストが失敗することを確認する**まで次に進まない
- 「正しい理由で失敗していること」を確認する（コンパイルエラーで失敗していないか、想定外のエラーで失敗していないか）

### 1 サイクルにつき 1 テスト

複数テストを先に書かない。1つ書いて Red → Green → Refactor を回し、次のテストへ。

---

## Phase 3: Green — テストを「本当に」成功させる

書いたテストを通すための **最小限の実装** だけを書く。この段階では美しさよりも「テストを緑にする」ことを優先。

### 厳禁

- アサーションの削除や緩和
- テストの `skip` / `comment-out`
- 「とりあえず動くようにする」ためのテストの書き換え

「本当に成功させる」こと。誤魔化しは TDD のフィードバックを破壊する。

### 実装の 3 つのギア（使い分け）

t-wada 流。状況に応じてペースを調整する:

| ギア | いつ使うか | やり方 |
|---|---|---|
| **仮実装（Fake It）** | 一般解への自信が無いとき、最初の一歩 | ハードコードでテストを通す（例: `return 42`） |
| **三角測量（Triangulation）** | 仮実装の後、一般化を強制したいとき | 別のテストケースを書き、ハードコードでは通らないようにすることで一般解を引き出す |
| **明白な実装（Obvious Implementation）** | 一般解への自信が高く、テストとの結びつきが明確なとき | いきなり一般化された実装を書く（仮実装を経由しない） |

> **判断基準**: 自信が高ければ「明白な実装」、低ければ「仮実装 → 三角測量」。迷ったときは仮実装から入る。

---

## Phase 4: Refactor — テストが緑のうちに設計を改善する

テストが緑のうちにだけ、コードの構造を改善する。

### やること

- 重複コードの抽出
- 命名の改善（変数・関数・型）
- 責務の分離（過度に大きな関数を分解）
- マジックナンバー・マジック文字列の定数化

### やらないこと

- **過度なリファクタリング**: 今のテストでカバーされていない設計変更
- **早すぎる抽象化**: 「将来必要になるかも」で抽象を導入
- **テストを赤にする変更**: テストが落ちた瞬間に戻す

### Refactor 判断チェックリスト

| 質問 | Yes なら refactor 候補 |
|---|---|
| 同じロジックが2箇所以上に出現するか | ✅ |
| 関数・変数の名前が振る舞いを表していないか | ✅ |
| 関数が複数の責務を負っていないか | ✅ |
| マジック値が散らばっていないか | ✅ |

すべて No なら、リファクタはスキップして次のテストへ進む。

---

## Phase 5: 繰り返し

テストリストが空になるまで Phase 2〜4 を繰り返す。途中で気付いた観点はリストに追加してよい（実装後に追加するのは禁止）。

### 全テストの実行

各サイクル終了時に、**既存テストを含むすべてのテストが緑** であることを確認する。1テストだけパスして他が落ちている状態でサイクルを進めない。

---

## 既存変更・バグ修正時の追加配慮

既存コードの変更や、バグ修正の場合は次を必ず守る:

- **変更前に既存テストが全て通る**ことを確認する（基準点の固定）
- **バグ修正の場合**: バグを **再現する失敗テストを先に書く**（Red Phase の延長）。バグ修正の妥当性はテストが緑になることで示す
- **変更後にも既存テストが引き続き通る**ことを確認する（回帰防止）
- **後方互換性が壊れる場合** は実装を進める前に呼び出し元へ申告し、判断を仰ぐ

---

## テストコードのガイドライン

- プロジェクトの **テストフレームワーク・既存パターン** に従う（CLAUDE.md と既存テストを参照）
- **粒度を分けて書く**:
  - 単体テスト: 純粋関数・小さな単位の振る舞い
  - 結合テスト: 複数モジュールの協調・外部 I/O 経由のフロー
- **モックの方針**:
  - モックしてよい: 外部サービス（API・メール・支払い）、認証、インフラ層（DB ドライバ・ファイルシステム）
  - モックしてはいけない: ビジネスロジック・ドメインモデル・テスト対象自身の主要協力子
- **1テスト1検証**: 1つのテストは1つの振る舞いだけを検証する
- **テストの独立性**: テストは順序非依存で実行できるようにする
- **動く仕様書として書く**: テスト名がそのまま振る舞いの説明になるようにする

## LLM/AI 実装での落とし穴と対策

AI が TDD を実行するとき特有の罠と、その回避策:

| 罠 | 対策 |
|---|---|
| **実装ファーストに傾く**（テストより先に実装を書こうとする） | Phase 2 を必ず通す。「テストが赤」を観測するまで実装ファイルを開かない |
| **複数テストをまとめて書く** | 「1サイクルにつき1テスト」を強制。次のテストはサイクル完了後に追加する |
| **Green を急いで誤魔化す** | アサーション緩和・テストスキップを禁止。仮実装は OK だが、必ず三角測量で一般化する |
| **Refactor を飛ばす** | サイクル毎に必ず Refactor チェックリストを通る（やらない判断も明示する） |
| **テストリストを途中で軽視** | 追加観点は実装前に必ずリストへ書く |

---

## 完了条件

このスキルの完了条件:

- テストリストのすべての項目が「対応するテスト」になっている
- すべてのテストが緑である
- 既存テストを壊していない
- 仕様（完了条件・受入基準）をテストレベルで満たしている

> Lint・型チェック・全テスト一括実行（`/quality-check`）はこのスキルの範囲外。完了後に呼び出し元が必ず実行する。

## 返却内容

呼び出し元へ次を返す:

- **テストリスト**（最終版。完了状況のチェックマーク付き）
- **追加・変更したファイル一覧**（テスト / プロダクションコード別）
- **TDD サイクルの履歴**: 各サイクルで使ったギア（仮実装 / 三角測量 / 明白な実装）と理由
- **既存テストの実行状況**（変更前 / 変更後）
- **後方互換性に影響がある場合**はその内容と判断材料
