# Self Review

現在のブランチの変更差分に対してセルフレビューを実施します。

## 手順

### 1. 変更差分の取得

差分の基準ブランチを動的に解決する（`main` 決め打ちにしない。既定ブランチが `master`/`develop` 等のリポジトリや、統合ブランチ方式で base が `feat/issue-*` の場合に、兄弟チケットの差分混入・コマンド失敗を防ぐため）:

```bash
# base の決定: 呼び出し元から渡されていればそれを優先。無ければ PR の base → リポジトリの既定ブランチ
BASE=$(gh pr view --json baseRefName -q '.baseRefName' 2>/dev/null \
  || gh repo view --json defaultBranchRef -q '.defaultBranchRef.name')
git log "origin/$BASE..HEAD" --oneline
git diff "origin/$BASE...HEAD"
```

### 2. コードレビューの実施

`code-reviewer` エージェントに委譲してレビューを実施（差分の読み込みと指摘の推論をサブエージェントの隔離コンテキストで行い、メインには結果のみを返すことでコンテキストを節約する）:

- コーディング規約準拠
- 品質・セキュリティ
- パフォーマンス（過剰最適化を避け、ボトルネック性とコストをケースバイケースで判断）
- テスト充足
- E2Eテスト（対象機能の場合）
- クロスリポジトリ依存仮定の確証（実装が他リポジトリの API の副作用・導出・判定ロジックに依存する場合、依存先リポジトリの実コードでの確証根拠 repo@ref/file:line（コミット SHA 等でリビジョン固定）があるか。**自リポジトリのモックテスト green は仮定の検証とみなさない**。根拠が無ければ「未確証の仮定」として指摘する）

### 3. 設計レビューの実施

`design-reviewer` エージェントに委譲して設計レビューを実施:

- 依存方向の検証
- パッケージ境界の検証
- Feature-based 構造の検証
- プロジェクト固有の設計検証

### 4. レビュー結果の報告

以下の形式で報告:

1. **問題点**: 修正が必要な箇所
2. **改善提案**: より良い実装の提案（任意）
3. **良い点**: 評価できる実装（任意）

### 5. 問題がある場合

指摘事項を修正し、再度セルフレビューを実施。すべての問題が解消されるまで繰り返す。
