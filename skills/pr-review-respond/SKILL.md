# PRレビュー対応

PR に付いた **AIレビュー（CodeRabbit 等）と人間レビューの両方** に対応するスキル。コメント取得 → 分類 → 修正 → 必須ゲート → コミット → 返信＆Resolved化 まで担当する。

> **前提条件**: GitHub CLI が設定済み。`para-impl` の続きとしても、独立した PR への対応としても呼び出せる。

---

## 入力

PR番号（省略可能）: $ARGUMENTS

省略時は現在のブランチに紐づく PR を `gh pr view --json number` で自動特定する。見つからない場合はユーザーに PR 番号の指定を求める。

---

## Phase 1: コメントの包括的取得

レビューコメントは 3 経路に分かれている。**いずれも漏らさない**:

```bash
# 1. レビュー本体（approve / changes_requested + 全体コメント）
gh pr view {PR番号} --json reviews

# 2. PR会話タブのコメント（個別行に紐づかないもの）
gh pr view {PR番号} --json comments

# 3. インラインコメント（個別の行に紐づくレビューコメント）
gh api repos/{owner}/{repo}/pulls/{PR番号}/comments
```

`{owner}/{repo}` は `gh repo view --json owner,name` で取得できる。

CodeRabbit 等の AI レビュアーのコメントもこれら 3 経路で網羅される。

---

## Phase 2: 分類

取得したコメントを以下に分類する:

| 分類 | 内容 | 対応 |
|---|---|---|
| **即時対応** | 明らかなバグ、規約違反、タイポ、命名、軽微なリファクタ | AI 判断で修正実施 |
| **設計変更** | アーキテクチャ・データモデル・API契約への影響あり | **ユーザーに確認**してから対応 |
| **クリティカル箇所** | 認証/認可・機密データ・外部連携・DBスキーマ等を含む | **ユーザーレビュー必須**（戦略§3.4） |
| **スコープ拡大** | 当該 PR のスコープ外の改善提案 | 不採用が原則、別 Issue 化を提案 |
| **不採用候補** | 提案が合理的でない、または既に対応済み | 理由を添えて返信のみ |
| **質問** | 設計意図・実装判断の説明依頼 | 返信のみ、コード変更なし |

各コメントの分類と対応方針を1つの表にまとめ、即時対応とユーザー確認待ちに分けて整理する。

---

## Phase 3: 修正と必須ゲート

「即時対応」分類のコメントについて修正を実施する。

1. 該当箇所を確認し、適切な修正を行う
2. 関連テストがあれば更新（バグ修正の場合は**再現テスト先行**）
3. 修正完了後、**`/quality-check` を実行**し、機械可読な結果が `pass` であることを確認する
4. 失敗時は修正して再実行。3回反復しても通らなければユーザーに判断を仰ぐ

> 必須ゲート未通過のコードはコミットしない。

「設計変更」「クリティカル箇所」のコメントは、ユーザー承認後に同じ手順で実施する。

---

## Phase 4: コミット

`/commit` スキルを呼び出してコミット規約に従ったコミットを作成する。

> `/commit` は内部で safety net としての `/quality-check` を再度回す。Phase 3 で通過済みなら速やかに完了する。

複数の独立した修正を行った場合は、論理的な単位ごとにコミットを分ける。

プッシュ:

```bash
git push
```

---

## Phase 5: 返信と Resolved 化

### 個別インラインコメントへの返信＆Resolved 化

```bash
# 返信
gh api -X POST repos/{owner}/{repo}/pulls/{PR番号}/comments \
  -f body="対応しました: {対応内容}" \
  -F in_reply_to={コメントID}

# Resolved 化（GraphQL）
gh api graphql -f query='
  mutation($threadId:ID!) {
    resolveReviewThread(input:{threadId:$threadId}) {
      thread { isResolved }
    }
  }' -F threadId={スレッドID}
```

スレッドIDは `gh api graphql` で `repository.pullRequest.reviewThreads` を辿って取得する。

### 不採用コメントへの返信

不採用にしたコメントには **理由を添えて返信**する（無視しない）:

```text
ご指摘ありがとうございます。今回はスコープ外と判断したため、別 Issue として #XXX に切り出しました。
```

### サマリーコメント

PR 全体に対するサマリーを最後に投稿する:

```bash
gh pr comment {PR番号} --body "$(cat <<'EOF'
レビュー対応しました。

## 対応内容
- {対応した項目1}
- {対応した項目2}

## 不採用・別Issue化
- {コメント要約}: {理由 or 別Issue番号}

ご確認をお願いします。
EOF
)"
```

---

## 完了報告

作業完了時に以下を報告する:

1. 対応したコメント数（即時対応 / 設計変更 / クリティカル / 不採用 / 質問）
2. 修正したファイル一覧と概要
3. `/quality-check` の最終結果
4. Resolved 化したスレッド数 / 未 Resolved のもの（人間レビュアーの確認待ち等）
5. 別 Issue 化したスコープ外指摘（あれば）

---

## ユーザーへの確認タイミング

- レビュー指摘の意図が不明確
- 設計変更を伴う指摘
- クリティカル箇所（戦略§3.4）に該当する指摘
- 対応コストが高い修正依頼
- 指摘内容に同意できない場合
- `/quality-check` が3回反復しても `pass` にならない場合
