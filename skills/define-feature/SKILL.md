# 機能定義（feature definition）

ユーザーとの対話を通じて機能を具体化し、**機能仕様ドキュメント (`docs/features/{slug}.md`) を作成**します。要件・クリティカル設計決定・必要に応じて機能全体の設計（アーキ・IF・データモデル）を **1ドキュメントに集約**します。

---

## 入力

テーマや概要: $ARGUMENTS

---

## 手順

### 1. プロジェクト理解

`CLAUDE.md` を読み、以下を把握:

- 技術スタック・ディレクトリ構成
- 既存機能・モジュール
- ドキュメントの配置ルール（`docs/` 配下の構造、ドキュメントマップ）
- 設計規約・コーディング規約

### 2. 要件ヒアリング

`$ARGUMENTS` が指定されている場合はそれをテーマとして使用し、不足情報をユーザーに確認する。指定がない場合はユーザーに要件のテーマを質問する。

以下の観点を対話的に確認（すべて必須ではなく、機能に応じて取捨選択する）:

- **何を作りたいか**: 機能の概要
- **なぜ必要か**: 背景・目的
- **誰が使うか**: ユーザーストーリー
- **何ができればよいか**: 機能要件（具体的な振る舞い）
- **制約はあるか**: 非機能要件（パフォーマンス、セキュリティ等）、技術的制約
- **画面・APIはあるか**: UI/APIの設計指針（該当する場合）
- **どうなれば完了か**: 受入基準

> **ヒアリングのコツ**: 一度に全項目を聞くのではなく、概要→詳細の順で対話的に深掘りする。ユーザーの回答から推測できる項目は提案して確認を取る。

### 3. クリティカル設計の調査・意思決定

要件に**クリティカル領域**が含まれるかを判定し、含まれる場合は **このフェーズで意思決定までまとめる**。実装フェーズに人間ゲートを残さないことで、後続の `/para-impl` で複数チケットを中断なく並列実行できる。

#### 3-1. クリティカル領域の識別

要件から次の領域への影響有無を確認する:

| 領域 | 例 |
|------|-----|
| セキュリティ | 認証フロー、認可ポリシー、入力バリデーション |
| 機密データ・個人情報 | 個人情報の保管・転送・ログ出力ポリシー |
| 外部システム連携 | 外部API契約、リトライ・タイムアウト方針、リクエスト署名 |
| DBスキーマ変更 | テーブル追加・カラム変更・インデックス設計・データ移行 |

該当しない場合はこのステップをスキップする。

#### 3-2. 既存パターン・制約の調査

該当領域が **1つのみ** の場合は並列化のメリットがないため、下記の Task 並列起動は行わず従来どおり直接調査する:

- 既存の認可モデル / 暗号化方式 / 外部APIクライアント / マイグレーション方式 など
- 関連する設計ドキュメント（CLAUDE.md ドキュメントマップから辿る）
- フレームワークの推奨パターン・制約

該当領域が **複数** の場合は、領域ごとに Task（`subagent_type: general-purpose`, `model: sonnet`）を **1メッセージで並列起動**して fan-out 調査させる。各サブエージェントへのプロンプトには以下を渡す:

- 調査対象の領域名（例: 認可モデル）
- **当該領域に関係する部分のみを抜粋した要件サマリ**（ヒアリング全文は渡さない）
- CLAUDE.md のドキュメントマップ・関連設計ドキュメントのパス（サブエージェントがコードベースと合わせて Glob/Grep/Read で調査できるように）
- 報告フォーマットの指示:
  - `options`: 選択肢を**最低2案**、各案の pros/cons
  - `pattern_alignment`: 既存パターンとの整合評価
  - `recommended`: 推奨案とその理由
  - `existing_patterns`: 参照した既存パターン・ドキュメントを path + 1行要約で**最大5件**

全サブエージェントの報告が揃ったら、メイン（opus/xhigh）エージェントはヒアリングの全文脈を踏まえて各サブエージェントの `recommended`（推奨案）を検証し、必要なら修正する。各サブエージェントの報告をそのまま次のステップの提示に流し込まない。複数領域が相互作用する場合（例: 機密データ×DBスキーマ）は、各サブエージェントの調査結果をメインが統合してから次のステップへ渡す。

#### 3-3. 設計選択肢の提示と意思決定

ユーザーに**選択肢と推奨案**を提示し、決定する:

```text
## クリティカル設計の意思決定

### {領域名 例: 認可モデル}

**選択肢**:
- A: {案A} — {pros/cons}
- B: {案B} — {pros/cons}

**推奨**: {案X}（理由: {既存パターンとの整合・実装コスト・拡張性 等}）

採用案を選んでください。
```

ユーザーが選んだ案を確定する。必要なら複数領域について順に決定する。

### 4. 機能全体の設計（必要な場合のみ）

機能が複数チケットにまたがる、または非自明なアーキ判断・IF/API・データモデルを含む場合は、コードベース分析を踏まえて以下を起草する。シンプルな機能ではスキップしてよい。

- **アーキ決定**: 実装方針と理由、変更対象モジュール、既存コードとの関係
- **IF / API**: モジュール間の境界となるインターフェース契約
- **データモデル**: 非クリティカルなデータ構造（クリティカルなスキーマ変更はステップ3で決定済み）
- **実装計画**: チケット分解の見通し（あくまで案。最終的な分解は `/create-ticket` で行う）

要件と同じドキュメント内に「## 機能全体の設計」として記述する。

### 5. テンプレート読み込み

以下の優先順位でテンプレートを読み込む:

1. `CLAUDE.md` に機能仕様テンプレートのパスが指定されている場合はそれ
2. なければ `skills/define-feature/templates/feature-spec.md`

### 6. 機能仕様ドキュメント作成

テンプレートに沿って機能仕様ドキュメントを作成する。

- **保存先**: `docs/features/{slug}.md`
  - `CLAUDE.md` に別パスが指定されていればそれに従う
  - `{slug}` は機能名をケバブケースにしたもの（例: `user-authentication`）
- **内容**: ヒアリング結果（要件）＋クリティカル設計決定＋（あれば）機能全体の設計を1ドキュメントに統合
- 該当しないセクションは**セクションごと削除**する（埋めるための水増しは禁止）
- 機能要件・受入基準はチェックボックス形式で記述する

### 7. 完了報告

```text
## 機能定義 完了

- 機能仕様ドキュメント: docs/features/{slug}.md

次のステップ:
# 1. 要件チケット(GitHub Issue) を作成
/create-ticket docs/features/{slug}.md

# 2. 実装チケットに分解
/create-ticket <作成された親Issue番号>

# 3. 実装チケットが揃ったら並列実装
/para-impl <実装チケット番号...>
```

## このスキルの責務外

- GitHub Issue 作成（`/create-ticket`）
- 実装計画のチケット粒度展開（`/create-ticket` の実装分解モード）
- TDD実装（`feature-implementer`）
- 個別の設計判断記録（ADR が必要なら別スキル）
